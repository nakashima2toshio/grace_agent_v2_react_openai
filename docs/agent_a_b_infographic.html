系統A: ReAct + Reflection  vs  系統B: GRACE 動的Agent

2種類のRAGエージェント機構を比較 — anthropic_grace_agent_v2

共通エントリ（左メニューで切替）

uv run streamlit run agent_rag.py --server.port 8501
🤖
系統A

自立型 Agent（ReAct + Reflection）agent_chat → ReActAgent（services/agent_service.py）

LLM が「思考→行動→観察」を反復し、毎ターン自分でツール要否を判断。最後に Reflection で推敲する2段構成。
制御方式LLM主導（毎ターン判断）
ツール定義_build_tools() に dict スキーマ直書き（input_schema）
  1. 入力拡張：MeCab で重要キーワード注入
  2. generate_with_tools() で LLM に問い合わせ
  3. stop_reason == tool_use ならツール実行 → 履歴に追記
↑ tool_use の間は②へループ（最大 max_turns）／ end_turn で抜ける
  1. Phase2 Reflection：tools=[] で自己評価・推敲 → Final Answer
⚡ 動的の正体：「ループを続けるか・どのツールをどのクエリで呼ぶか」をLLMが毎ターン決める（構造 ReAct→Reflection は固定）
🧭
系統B ［最新］

自律型 Agent（GRACE / Plan + Execute）grace_chat → Planner + Executor（grace/）

先に実行計画を立て、1ステップずつ実行しながら、結果に応じてフォールバック・SKIP・リプラン・人間介入を行う自律適応型。
制御方式計画主導（Plan→Execute）＋実行時適応
ツール定義BaseTool クラス＋ToolRegistry 登録
  1. Plan：複雑度推定 → 単純=ルールベース / 複雑=LLM計画
  2. Execute：依存チェック＋並列先行検索 → ツール実行 → 信頼度算出
  3. 動的フォールバック：RAG不足なら web_search、失敗なら ask_user／十分なら SKIP
  4. 動的介入：信頼度で SILENT / NOTIFY / CONFIRM / ESCALATE
  5. 動的リプラン：失敗・低信頼度で計画を作り直す（最大3回）
⚡ 動的の正体：「どの経路を通り、失敗したらどう立て直すか」をコードのルール（スコア / 成功失敗 / 信頼度）が状況適応で決める

📊 処理の比較・違い

観点系統A — ReAct+Reflection系統B — GRACE 動的Agent
制御方式LLM主導（毎ターン判断）計画主導（Plan→Execute）＋実行時適応
事前計画なし（その場で判断）あり（複雑度で rule / LLM 切替）
判断主体LLMコードのルール（一部 LLM 補助）
「動的」の正体ループ継続/終了・ツール要否（stop_reason）経路切替・SKIP・リプラン・介入
フォールバックなしあり（rag→web→ask_user）
失敗時のやり直しなし（max_turns で打ち切り）あり（リプラン最大3回）
Human-in-the-loopなしあり（CONFIRM / ESCALATE）
並列検索なし（逐次）あり（先行プリフェッチ）
品質担保Reflection（自己評価・推敲）多軸信頼度 ＋ リプラン ＋ 介入
透明性表示Thought / ツール呼び出しログ計画(Plan) ＋ 実行ログ(Execute) ＋ 複雑度
レイテンシ/コスト中 ターン数依存低〜高 単純=LLMゼロ / 複雑=計画+リプラン

🎯 使い分けの指針

系統A が向くケース

  • 単発の Q&A、検索→回答が素直なタスク
  • 思考過程（Thought）を見せたいデモ
  • 軽量・素早い応答を優先したいとき

系統B が向くケース

  • 複数ステップ・複数情報源を跨ぐ調査
  • 検索失敗時に Web / ユーザー確認へ自動切替したい
  • 信頼度に応じた人間介入、計画と実行の可視化が欲しい
🔗 関係：系統A は基盤。系統B は実行中に run_legacy_agent ステップで系統Aを1手段として内包しうる（上位）。
